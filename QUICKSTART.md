## 5分でわかる概要

### DORAとは？
DORA（DevOps Research and Assessment）は、Google Cloudが運営する研究プログラムで、ソフトウェアデリバリーとオペレーションのパフォーマンスを向上させる能力（Capabilities）を特定しています。

### このフレームワークの目的
組織のDevOps成熟度を**5段階**で評価し、**具体的な改善計画**を立てるためのフレームワークです。

### 5つの成熟度レベル
1. **レベル1: 初期段階** - アドホックなプロセス
2. **レベル2: 管理段階** - 基本的な管理とプロセス
3. **レベル3: 定義段階** - 組織標準として確立
4. **レベル4: 定量管理段階** - データドリブンな管理
5. **レベル5: 最適化段階** - 継続的改善と革新

### 対象となる27のケイパビリティ
- **学習のための風土**: 9ケイパビリティ
- **高速フロー**: 9ケイパビリティ
- **高速フィードバック**: 9ケイパビリティ

## 最初の評価（60分）

ここでは **Continuous Integration** を例に、評価の手順を説明します。

### ステップ1: ケイパビリティの理解（10分）

1. **HTMLファイルを読む**
   ```
   capabilities/DORA _ Capabilities_ Continuous Integration.html
   ```
   ブラウザで開き、ケイパビリティの定義と実装方法を理解します。

2. **成熟度モデルを読む**
   ```
   maturity-models/continuous-integration.md
   ```
   5つのレベルの違いを理解します。

### ステップ2: 評価シートの作成（5分）

1. **evaluationsディレクトリを作成**
   ```bash
   mkdir -p evaluations
   ```

2. **テンプレートをコピー**
   ```bash
   cp maturity-models/EVALUATION_TEMPLATE.md \
      evaluations/continuous-integration-2025-11.md
   ```

3. **基本情報を記入**
   - 評価日
   - 評価者
   - 評価対象（チーム/部門名）

### ステップ3: データ収集（20分）

#### 定量データの収集

CIの場合、以下の情報を集めます：

- [ ] 統合頻度（コミット頻度）
- [ ] ビルド実行頻度
- [ ] ビルド成功率
- [ ] ビルド時間（平均・最大）
- [ ] テストカバレッジ
- [ ] ブランチの平均寿命

**収集方法の例**:
```bash
# Git統計
git log --since="1 month ago" --oneline | wc -l    # コミット数
git branch -r | wc -l                               # ブランチ数

# CIツールのダッシュボードから:
# - ビルド成功率
# - 平均ビルド時間
# - テスト実行結果
```

#### 定性データの収集

簡単なインタビューまたはアンケート：

**質問例**:
1. 「コードを統合する頻度はどれくらいですか？」
2. 「ビルドが壊れたとき、どのくらいの優先度で修正しますか？」
3. 「テストは自動実行されていますか？」
4. 「ビルド時間について満足していますか？」

### ステップ4: レベル判定（15分）

評価シートの各項目をチェックします：

#### 実行頻度
```markdown
- [ ] 週1回以下                → レベル1
- [ ] 日次〜週に数回           → レベル2
- [ ] 日次〜複数回/日          → レベル3
- [ ] 複数回/日、オンデマンド   → レベル4
- [ ] 継続的、フローベース     → レベル5
```

#### 自動化レベル
```markdown
- [ ] 手動プロセス                  → レベル1
- [ ] 一部自動化                    → レベル2
- [ ] 主要プロセス自動化            → レベル3
- [ ] ほぼ完全自動化                → レベル4
- [ ] 完全自動化 + AI/ML           → レベル5
```

**判定ルール**: 最も多くの基準を満たしているレベルを選択

### ステップ5: 結果のまとめ（10分）

評価シートの以下のセクションを記入：

1. **エグゼクティブサマリー**
   - 現在のレベル
   - 主な発見事項（3つ）
   - 推奨アクション（上位3つ）

2. **SWOT分析**
   - 強み・弱み・機会・脅威を各3つ

**記入例**:
```markdown
## エグゼクティブサマリー

### 現在の成熟度レベル
**レベル2: 管理段階**

### 主な発見事項
- CIツール（Jenkins）は導入済みだが、日次ビルドのみ
- テストカバレッジは35%と低い
- ビルド失敗時の対応プロセスが不明確

### 推奨アクション（上位3つ）
1. コミットごとのビルド実行に変更
2. テストカバレッジを70%まで向上
3. ビルド失敗対応プロセスの文書化と周知
```

## 改善計画の作成（30分）

### ステップ1: 目標設定（5分）

**SMART目標**を設定します：

- **S**pecific（具体的）
- **M**easurable（測定可能）
- **A**chievable（達成可能）
- **R**elevant（関連性）
- **T**ime-bound（期限付き）

**例**:
```markdown
目標: 3ヶ月以内にレベル3（定義段階）に到達
- コミットごとのビルド・テスト実行
- ビルド時間 < 10分
- テストカバレッジ > 70%
```

### ステップ2: アクションの特定（15分）

現在のレベルと目標レベルのギャップから、必要なアクションを特定します。

**レベル2→3の典型的なアクション**:

| アクション | 優先度 | 期間 | 難易度 |
|-----------|--------|------|--------|
| コミットフックの設定 | 高 | 1週間 | 低 |
| テストカバレッジ向上 | 高 | 2ヶ月 | 中 |
| ビルド時間最適化 | 中 | 1ヶ月 | 中 |
| トランクベース開発への移行 | 高 | 3ヶ月 | 高 |
| ビルド失敗対応ルール策定 | 高 | 2週間 | 低 |

### ステップ3: スケジュール作成（10分）

**ガントチャート形式**でスケジュールを作成：

```
月     |1  |2  |3  |4  |5  |6  |
-------|---|---|---|---|---|---|
コミットフック設定
       [==]
テストカバレッジ向上
       [============]
ビルド最適化
           [======]
トランクベース移行
               [==========]
定期レビュー
       △   △   △   △   △   △
```

**クイックウィンを最初に**: 早い段階で小さな成功を積み重ねることが重要です。

## よくある質問（FAQ）

### Q1: どのケイパビリティから始めるべき？

**A:** 以下の基準で選択してください：

1. **現在の課題が明確** → その課題に関連するケイパビリティ
2. **技術チームが主導** → Continuous Integration, Version Control
3. **マネジメントが主導** → Learning Culture, Transformational Leadership
4. **迷ったら** → INDEX.mdの「優先度別推奨順序」を参照

### Q2: 評価には誰を巻き込むべき？

**A:** 理想的な評価チーム構成：

- **必須**: 実際にプラクティスを実行しているエンジニア 2-3名
- **推奨**: チームリーダー/マネージャー 1名
- **任意**: 他チームのメンバー（客観的視点）1名

### Q3: すべてレベル5を目指すべき？

**A:** いいえ。**コンテキストに応じた最適なレベル**を目指してください：

- **スタートアップ**: レベル2-3で十分な場合も
- **エンタープライズ**: レベル3-4が現実的な目標
- **テックカンパニー**: レベル4-5を目指す価値あり

**ROI（投資対効果）**を考慮して目標設定しましょう。

### Q4: 評価にどのくらい時間がかかる？

**A:** 初回評価の目安：

- **簡易評価**: 1-2時間（スコア付けのみ）
- **標準評価**: 半日（データ収集含む）
- **包括的評価**: 1-2日（インタビュー、分析含む）

**2回目以降は短縮**されます（前回との差分のみ）。

### Q5: 評価結果を経営層に報告するには？

**A:** エグゼクティブサマリーを使用：

```markdown
# Continuous Integration 成熟度評価サマリー

## 現状
- **レベル2/5**: 基本的なCIは導入済み
- **主な課題**: 統合頻度が低く、品質問題の早期発見が困難

## ビジネス影響
- リリース遅延: 平均2日/月（推定コスト: XXX万円）
- 障害対応時間: 平均4時間（推定コスト: XX万円）

## 改善提案
- **投資**: XXX万円（ツール、トレーニング）
- **期待効果**:
  - リリースサイクル 30%短縮
  - 障害対応時間 50%削減
- **ROI**: 6ヶ月で回収見込み

## 承認事項
1. 予算承認: XXX万円
2. リソース配分: エンジニア1名、3ヶ月
```

## チートシート

### 評価時のチェックポイント

```markdown
□ データを集める（主観だけに頼らない）
□ 複数人の意見を聞く（バイアスを避ける）
□ エビデンスを保存する（スクリーンショット、ログ等）
□ 強みも弱みも正直に記録する
□ 改善可能な具体的アクションを特定する
```

### レベル判定の目安

| レベル | キーワード | 典型的な状態 |
|--------|----------|-------------|
| 1 | アドホック、手動 | 「私の環境では動く」 |
| 2 | 基本的、プロジェクト | 「このチームではやっている」 |
| 3 | 標準化、組織 | 「全社で統一されている」 |
| 4 | 測定、データ | 「数字で管理している」 |
| 5 | 最適化、革新 | 「業界をリードしている」 |

### 優先順位の付け方

```
優先度 = (ビジネス影響度 × 実現可能性) / 必要コスト

ビジネス影響度: 1-5点
実現可能性: 1-5点
必要コスト: 相対評価（1-5点）
```

## 成功のためのヒント

### ✅ Do（推奨）

- **小さく始める**: 1つのケイパビリティから
- **データを集める**: 主観だけに頼らない
- **クイックウィンを作る**: 早期の成功体験
- **定期的にレビュー**: 月次または四半期
- **チーム を巻き込む**: 当事者意識を醸成
- **成功を共有**: モチベーション維持

### ❌ Don't（避けるべき）

- **完璧を目指す**: まず始めることが重要
- **一度にすべて**: 焦点がぼやける
- **評価のための評価**: アクションに繋げる
- **トップダウンのみ**: 現場の意見を無視
- **短期的視点**: 文化変革には時間が必要
- **失敗を責める**: 心理的安全性が重要
